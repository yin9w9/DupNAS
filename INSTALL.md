# Local Installation and Execution

This document describes how to reproduce DupNAS evaluations and MCU inference experiments using the provided Docker image.

Docker image:

```text
yin9w9/dupnas-ae:latest
```

Docker Hub:

```text
https://hub.docker.com/r/yin9w9/dupnas-ae
```

The image includes the DupNAS source code, required Python dependencies, pre-generated ONNX models, and deployable TensorFlow Lite models.

---

## 1. DupNAS Local Setup

### 1.1 Requirements

* Linux host
* Docker
* NVIDIA Container Toolkit
* NVIDIA GPU for training or fine-tuning
* Kaggle API token for downloading ImageNet-100

### 1.2 Start the Container

```bash
sudo docker run -it \
  --name dupnas \
  --gpus all \
  yin9w9/dupnas-ae:latest
```

Inside the container, navigate to the DupNAS directory:

```bash
cd DupNAS/
```

### 1.3 Configure ImageNet-100 Access

Create a Kaggle API token from:

```text
https://www.kaggle.com/datasets/ambityga/imagenet100
```

Then configure the token inside the container:

```bash
export KAGGLE_API_TOKEN="<your-token>"

mkdir -p ~/.kaggle
echo "<your-token>" > ~/.kaggle/access_token
chmod 600 ~/.kaggle/access_token
```

Download and prepare the dataset:

```bash
python3.9 download_image100.py
```

The Docker image contains the pre-generated models used in the paper. These models can be used directly for reproducing the evaluation results without rerunning the complete neural architecture search.

---

### 1.4 Peak-Memory Evaluation Example

The following example evaluates ShuffleNetV2 under the 96 KB VM constraint.

#### Run the Evaluation

```bash
cd sample_onnx/shufflenet

bash run_allsamples_vm96.sh ../outputs python3.9
```

#### Summarize the Results

```bash
cd ..

python3.9 analyze_all_output_dupnasa.py \
  --output_dir ./outputs \
  --model shufflenet \
  --vm 96
```

Detailed results are saved in:

```text
sample_onnx/outputs/
```

An example console summary is:

```text
[Summary] shufflenet vm96
original_feasible_ratio: 45.88%
dupnas_feasible_ratio: 24.62%
```

These results correspond to the peak-memory evaluation reported in Fig. 6 of the paper.

## 2. Inference Local Setup

The deployment workflow supports the following development boards:

* STM32F746G-DISCO
* STM32H747I-DISCO

The provided scripts support:

* TensorFlow Lite Micro
* STM32Cube.AI

### 2.1 Host Requirements

The following STM32 tools must be installed on an Ubuntu 24.04 LTS host:

* STM32Cube AI Studio 1.2.0
* STM32CubeMX 6.17.0
* STM32CubeIDE 2.1.1
* STM32CubeProgrammer
* GNU Arm Embedded Toolchain

The STM32 tools are assumed to be installed under:

```text
/opt/st
/opt/ST
```

Update the Docker mount paths and deployment scripts when different installation directories are used.

### 2.2 Identify the STM32 Devices

List connected STM32 boards:

```bash
STM32_Programmer_CLI -l
```

Identify the UART devices:

```bash
ls /dev/ttyACM*
```

Typical device paths are:

```text
/dev/ttyACM0
/dev/ttyACM1
```

The actual paths may differ depending on the connected USB devices.

### 2.3 Start the Deployment Container

```bash
sudo docker run -it \
  --name dupnas-deploy \
  --privileged \
  -v /dev/bus/usb:/dev/bus/usb \
  -v /opt/st:/opt/st \
  -v /opt/ST:/opt/ST \
  --device=/dev/ttyACM0 \
  --device=/dev/ttyACM1 \
  yin9w9/dupnas-ae:latest
```

Inside the container, navigate to the deployment scripts:

```bash
cd STM/scripts/
```

### 2.4 Configure the Deployment Scripts

Configure the scripts for the target board and inference engine:

```text
run_tflm_f7.sh
```

Update the following variables as required:

```text
PATH
PROGRAMMER
STEDGEAI
SERIAL_NUMBER
TTY_DEVICE
```

* `PATH`: GNU Arm toolchain directory.
* `PROGRAMMER`: path to `STM32_Programmer_CLI`.
* `STEDGEAI`: path to STM32Cube AI Studio. Required only for STM32Cube.AI.
* `SERIAL_NUMBER`: serial number reported by `STM32_Programmer_CLI -l`.
* `TTY_DEVICE`: UART device, such as `/dev/ttyACM0`.

---

### 2.5 Model Preparation

Pre-generated models are included in the following directories:

```text
TFLite models: STM/tflm-template/src/models/
ONNX models:   STM/onnx_models/
```

These models can be used directly for reproducing the deployment experiments.

#### Optional ONNX-to-TFLite Conversion

Models generated by a new DupNAS run must be converted into integer-quantized TensorFlow Lite models before TFLite Micro deployment.

```bash
cd Inference/onnx-to-tflite/
dockerd >/tmp/dockerd.log 2>&1 &
bash convert.sh shuffle
```

Converted models are saved under:

```text
Inference/onnx-to-tflite/outputs/
```

The generated filenames typically end with:

```text
_full_integer_quant.tflite
```

Copy the selected models to:

```text
STM/tflm-template/src/models/
```

---

### 2.6 Build the TFLite Micro Library

```bash
cd STM/tflm-template/

make
./tflm_main 2> host.txt
```

This builds and validates the TFLite Micro static library before MCU deployment.

---

### 2.7 Inference-Latency Example

The following example deploys one TFLite model to the STM32F746G-DISCO and measures its inference latency.

#### Create the Model List

```bash
cd STM/
```

create `models_tflite.txt` to contain:

```text
shuffle-im100-vm96-dupnas_w0.5_ir128_split_full_integer_quant.tflite
```
Each line must contain one model filename. The model must exist under:

```text
STM/tflm-template/src/models/
```

#### Patch the Repository Path

```bash
bash scripts/patch_repo_root.sh
```

#### Run the Latency Experiment

For STM32F746 with TFLite Micro:

```bash
bash scripts/run_tflm_f7.sh models_tflite.txt latency
```

The `latency` argument instructs the script to program the board and measure the inference latency of each listed model.

During MCU programming, the onboard LEDs may blink while the model is being flashed.

#### Results

For the STM32F746 TFLite Micro experiment, results are saved in:

```text
STM/results/tflm_f7/results.csv
```
